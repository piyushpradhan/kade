Notion-Driven AI Task Orchestration

A system for breaking prompts into tasks, tracking them in Notion, and auto-dispatching them to any AI provider's CLI agent — with zero token cost on database operations.

1. System Overview

The goal: you type a prompt into your CLI. An LLM breaks it into structured action items. Those items become tickets in a Notion database — complete with projects, goals, tasks, subtasks, dependencies, priorities, and a provider field specifying which AI agent should execute each task. A background daemon watches for tickets moved to in progress, dispatches them to the appropriate provider's CLI (Claude, Codex, OpenCode, Grok, Gemini, etc.), and updates the ticket state when the agent finishes.

Notion API (free) Multi-provider CLI dispatch Subtask hierarchy Dependency tracking Zero-token CRUD Polling-based (no webhooks needed)

What costs tokens

Only the initial prompt breakdown (in your existing CLI session) and the actual task execution by the provider agent. These are unavoidable — you'd pay for them anyway.

What is completely free

Populating Notion, polling for status changes, dispatching to a provider CLI, and updating ticket state. All done by deterministic scripts calling the Notion REST API directly — no LLM involved.

2. The Core Principle: Why This Is Cheap

Key Insight

Notion's REST API is free (rate-limited to ~3 requests/second, no monetary cost). The Notion MCP is just a wrapper around that API — the protocol is free, but the LLM driving the MCP burns tokens. If you bypass the LLM and call the API from scripts, every Notion operation becomes free.

The original idea was "use a cheap AI to populate Notion." The refinement: don't use any AI for Notion CRUD. If the breakdown step outputs structured JSON, a script can parse it and make direct API calls. No LLM, no MCP, no tokens.

The "cheap AI" approach (using GPT-4-mini, Haiku, or Gemini Flash to drive Notion MCP) is a fallback if you can't get structured JSON from the breakdown step — but it's strictly worse than the script approach. It costs *some* tokens and adds latency, with no benefit over deterministic code.

Approach Cost per Notion operation Latency Reliability
LLM driving Notion MCP Expensive Full LLM turn per operation 2-10 seconds LLM may hallucinate fields or misformat
Cheap LLM (Haiku/Flash) driving MCP Cheap but nonzero ~$0.001 per operation 1-3 seconds Same reliability risk, smaller
Script calling Notion API directly Free $0.00 always 50-200ms Deterministic, never fails on formatting

3. Architecture & Data Flow

1

You type a prompt in your CLI tokens: your session

Your normal LLM session (Claude, Codex, OpenCode, etc.) receives the prompt and breaks it into structured JSON: projects, tasks, subtasks, dependencies, priorities, and recommended provider per task.

↓
2

Notion Populator script tokens: $0

A Node.js/Python script reads the JSON and creates Notion pages in your database — tasks, subtasks (with parent relations), dependency relations, and all properties. Direct API calls, no LLM.

↓
3

You review in Notion UI tokens: $0

All tasks are visible in Notion. You can manually edit, reorder, reassign providers, add details, and move tasks from todo to in progress when you're ready for work to begin.

↓
4

Poller/Dispatcher daemon tokens: $0

A background process polls Notion every 30 seconds for tickets in in progress that aren't yet assigned. When it finds one, it reads the Provider property, marks the ticket as assigned (to prevent double-dispatch), and spawns the appropriate CLI agent as a subprocess.

↓
5

Provider agent executes the task tokens: provider session

The CLI agent (Claude, Codex, OpenCode, Grok, Gemini, etc.) works on the task in your repository. The task description from Notion is passed as the prompt. The agent has full codebase access.

↓
6

State updater script tokens: $0

When the agent subprocess exits, a wrapper script updates the Notion ticket: status → review, adds a timestamp, and optionally attaches the agent's output summary. Direct API call, no LLM.

4. Component Breakdown

Component 1: Prompt Breakdown tokens: your CLI session

This happens inside your existing CLI session. You type your prompt, and the LLM produces structured JSON. The key is prompting for JSON output, not free-form text. The LLM also assigns a provider to each task based on what's best suited (or you can set a default provider and override per-task in Notion later).

Example system prompt addition for your CLI:

// Add to your CLI agent's system prompt or AGENTS.md:

When the user asks you to plan or break down a task, output a JSON block
wrapped in <task-plan> tags. Structure:

{
  "project": "Project name",
  "goal": "One-line goal description",
  "tasks": [
    {
      "id": "t1",
      "title": "Task title",
      "description": "Detailed prompt for the agent",
      "provider": "codex",
      "priority": "high",
      "subtasks": [
        {
          "id": "t1.1",
          "title": "Subtask title",
          "description": "...",
          "provider": "opencode",
          "priority": "medium"
        }
      ],
      "blocked_by": ["t2"],
      "related": ["t3"]
    }
  ]
}

// Provider options: "claude", "codex", "opencode", "grok", "gemini"
// Priority options: "urgent", "high", "medium", "low"

Component 2: Notion Populator tokens: $0

A standalone script that takes the JSON from Component 1 and creates Notion pages. Run it manually after the breakdown, or pipe the JSON to it automatically.

Responsibilities:

  • Create a page for each task in the target Notion database
  • Set all properties (title, status, provider, priority, description)
  • Create parent-child relations for subtasks
  • Create dependency relations (blocked_by, related)
  • Set initial status to todo

Component 3: Poller / Dispatcher tokens: $0

A long-running daemon that watches the Notion database for tasks ready to be worked on.

Responsibilities:

  • Poll Notion every N seconds (default: 30s) for tasks where status = in_progress and assigned = false
  • For each found task: read the Provider property
  • Atomically mark the task as assigned = true (prevents double-dispatch if running multiple poller instances)
  • Spawn the appropriate CLI agent as a subprocess, passing the task description as the prompt
  • Track the subprocess; when it exits, trigger Component 4
  • Support concurrent task execution (configurable max parallelism)
  • Respect dependencies: don't dispatch a task if its blocked_by tasks aren't done

Component 4: State Updater tokens: $0

A function called by the dispatcher when an agent subprocess finishes.

Responsibilities:

  • Update the Notion ticket status to review (on success) or todo (on failure, with an error note)
  • Set assigned = false so it can be re-dispatched if moved back to in_progress
  • Record the completion timestamp
  • Optionally capture the agent's stdout/stderr and append it to the Notion page body

5. Notion Database Schema

Create a single Notion database (e.g., "Task Board") with the following properties. All of these are standard Notion property types — no custom integrations needed.

Property Name Notion Type Purpose
Name Title Task name / summary
Status Select Options: todo, in_progress, review, done, blocked
Provider Select Which CLI agent executes: claude, codex, opencode, grok, gemini
Priority Select Options: urgent, high, medium, low
Project Multi-select or Relation Group tasks by project/goal. Use Multi-select for simple grouping, or a separate Projects database with a Relation for richer metadata.
Parent Task Relation (self-referential) Links subtasks to their parent task. Create the relation, then filter to show only tasks where this is empty for top-level tasks.
Blocked By Relation (self-referential) Dependencies. The dispatcher checks that all blocked_by tasks are done before dispatching.
Related To Relation (self-referential) Non-blocking relationships between tasks (for reference / context sharing).
Assigned Checkbox Internal flag: true when the dispatcher has picked up the task. Prevents double-dispatch. Reset to false on completion.
Tags Multi-select Free-form categorization (e.g., bug, feature, refactor, docs)
Started At Date Timestamp when the agent began working
Completed At Date Timestamp when the agent finished
Exit Code Number The subprocess exit code (0 = success, nonzero = failure)
Error Log Text (or page body) Agent's stderr or error output, if the task failed

The task's detailed description / prompt for the agent goes in the page body (the rich text content area below the properties). This is what gets passed to the CLI agent as the prompt.

Notion Database Views to Create

  • Kanban Board — grouped by Status. This is your main view.
  • By Project — grouped by Project, filtered to Status ≠ done.
  • Queue (todo) — list view, filtered to Status = todo, sorted by Priority.
  • Active (in progress) — list view, filtered to Status = in_progress.
  • Review queue — list view, filtered to Status = review.
  • Subtasks — filtered to Parent Task is not empty.

6. Multi-Provider Dispatch

Each task has a Provider Select property. The dispatcher reads this and invokes the matching CLI. All major AI coding CLIs accept a prompt via a command-line flag and work in a specified directory.

Provider value in Notion CLI command Invocation pattern Notes
claude claude claude -p "task desc" --cwd /repo Claude Code CLI. Uses your Claude subscription.
codex codex codex --prompt "task desc" --cwd /repo OpenAI Codex CLI. Uses your OpenAI subscription.
opencode opencode opencode -p "task desc" --cwd /repo OpenCode CLI. Configurable model via opencode.json.
grok grok grok --prompt "task desc" --cwd /repo Grok CLI (if available). Adjust flags per your install.
gemini gemini gemini -p "task desc" --cwd /repo Gemini CLI. Uses Google AI credentials.
Provider Abstraction

Each provider CLI has slightly different flags. The dispatcher uses an adapter map (a simple JSON/dict config) that maps the Notion Provider value to the exact command template. Adding a new provider is a one-line config change — no code modification needed.

7. Token Cost Analysis

Step Uses LLM? Token cost Frequency
Prompt breakdown (Component 1) Yes — your CLI session Your session tokens Once per prompt
Populate Notion (Component 2) No — script $0.00 Once per prompt
Poll for in-progress tasks (Component 3) No — script $0.00 Every 30 seconds (Notion API, free)
Dispatch to provider CLI (Component 3) No — subprocess spawn $0.00 Once per task dispatch
Agent executes task (Component 3 → 5) Yes — provider agent Provider session tokens Once per task (unavoidable)
Update ticket state (Component 4) No — script $0.00 Once per task completion

Bottom line: You pay for the breakdown (which you're already doing) and the task execution (which is the actual work). All project management overhead — ticket creation, status tracking, dispatching, state updates — costs $0.00 in tokens.

9. Step-by-Step Implementation Plan

Phase 1: Notion Setup (manual, ~15 min)

  1. Create a Notion integration at notion.so/my-integrations → get the NOTION_TOKEN
  2. Create a new database in Notion with the 14 properties listed in Section 5
  3. Share the database with your integration (Share → invite your integration)
  4. Copy the database ID from the URL: notion.so/DATABASE_ID?v=...
  5. Create the views listed in Section 5 (Kanban, Queue, Active, Review, Subtasks)

Phase 2: Core Scripts (~2-3 hours)

  1. Initialize the project: npm init + npm install @notionhq/client
  2. Write notion-client.js — thin wrapper with createTask(), queryTasks(), updateTask(), createRelation()
  3. Write populate.js — reads JSON from stdin/file, creates pages + relations
  4. Write providers.js — provider adapter map + spawnAgent(provider, prompt, cwd) function
  5. Write updater.js — markComplete(taskId, exitCode, output) and markFailed(taskId, error)
  6. Write poller.js — the daemon loop: query → check dependencies → dispatch → await → update

Phase 3: Integration & Testing (~1 hour)

  1. Test populate: create a sample JSON, run node src/populate.js sample.json, verify pages appear in Notion
  2. Test dispatcher: manually move a task to in_progress in Notion, verify the poller picks it up and spawns the right CLI
  3. Test updater: verify the ticket status changes to review after the agent finishes
  4. Test dependencies: create a task blocked by another, verify it doesn't dispatch until the blocker is done
  5. Test concurrency: move multiple tasks to in_progress, verify they run in parallel (up to max concurrency)

Phase 4: Polish & Deployment (~30 min)

  1. Set up the daemon to run persistently: launchd (macOS), systemd (Linux), or tmux/screen
  2. Add logging to a file for debugging (logs/poller.log)
  3. Add a populate-from-stdin.sh script so you can pipe the breakdown JSON directly: claude -p "plan X" | node src/populate.js
  4. Optionally: add a CLI command kade that wraps the whole flow (prompt → breakdown → populate → confirm)

src/notion-client.js

const { Client } = require("@notionhq/client");

const config = require("../config.json");

const notion = new Client({
  auth: process.env[config.notion.token_env],
});

const DB_ID = config.notion.database_id;

async function createTask({ title, description, status, provider, priority, project, tags }) {
  const page = await notion.pages.create({
    parent: { database_id: DB_ID },
    properties: {
      Name: { title: [{ text: { content: title } }] },
      Status: { select: { name: status || "todo" } },
      Provider: { select: { name: provider || config.default_provider } },
      Priority: { select: { name: priority || "medium" } },
      ...(project && { Project: { multi_select: [{ name: project }] } }),
      ...(tags && { Tags: { multi_select: tags.map(t => ({ name: t })) } }),
      Assigned: { checkbox: false },
    },
    ...(description && {
      children: [{
        object: "block",
        type: "paragraph",
        paragraph: { rich_text: [{ text: { content: description } }] },
      }],
    }),
  });
  return page.id;
}

async function createSubtask(parentId, taskData) {
  const pageId = await createTask(taskData);
  await notion.pages.update({
    page_id: pageId,
    properties: {
      "Parent Task": { relation: [{ id: parentId }] },
    },
  });
  return pageId;
}

async function createRelation(pageId, propertyName, relatedPageId) {
  await notion.pages.update({
    page_id: pageId,
    properties: {
      [propertyName]: { relation: [{ id: relatedPageId }] },
    },
  });
}

async function queryInProgressUnassigned() {
  const response = await notion.databases.query({
    database_id: DB_ID,
    filter: {
      and: [
        { property: "Status", select: { equals: "in_progress" } },
        { property: "Assigned", checkbox: { equals: false } },
      ],
    },
  });
  return response.results;
}

async function getTaskDescription(pageId) {
  const blocks = await notion.blocks.children.list({ block_id: pageId });
  return blocks.results
    .map(b => b.paragraph?.rich_text?.map(t => t.plain_text).join("") || "")
    .join("\n");
}

async function markAssigned(pageId) {
  await notion.pages.update({
    page_id: pageId,
    properties: {
      Assigned: { checkbox: true },
      "Started At": { date: { start: new Date().toISOString() } },
    },
  });
}

async function markComplete(pageId, exitCode, output) {
  await notion.pages.update({
    page_id: pageId,
    properties: {
      Status: { select: { name: exitCode === 0 ? "review" : "todo" } },
      Assigned: { checkbox: false },
      "Completed At": { date: { start: new Date().toISOString() } },
      "Exit Code": { number: exitCode },
    },
  });

  if (output && exitCode !== 0) {
    await notion.blocks.children.append({
      block_id: pageId,
      children: [{
        object: "block",
        type: "code",
        code: { rich_text: [{ text: { content: output.slice(0, 2000) } }], language: "plain text" },
      }],
    });
  }
}

async function getBlockedByTasks(pageId) {
  const page = await notion.pages.retrieve({ page_id: pageId });
  const relations = page.properties["Blocked By"]?.relation || [];
  const blockedBy = [];
  for (const rel of relations) {
    const blockedPage = await notion.pages.retrieve({ page_id: rel.id });
    const status = blockedPage.properties["Status"].select?.name;
    blockedBy.push({ id: rel.id, status });
  }
  return blockedBy;
}

module.exports = {
  createTask, createSubtask, createRelation,
  queryInProgressUnassigned, getTaskDescription,
  markAssigned, markComplete, getBlockedByTasks,
};

src/populate.js

const { createTask, createSubtask, createRelation } = require("./notion-client");

/* Reads a task-plan JSON from stdin or a file argument, creates all pages + relations */
async function main() {
  let input;
  if (process.argv[2]) {
    input = require("fs").readFileSync(process.argv[2], "utf8");
  } else {
    input = await new Promise(res => {
      let data = "";
      process.stdin.on("data", chunk => data += chunk);
      process.stdin.on("end", () => res(data));
    });
  }

  /* Extract the JSON from <task-plan> tags if present, otherwise parse directly */
  const match = input.match(/<task-plan>([\s\S]*?)<\/task-plan>/);
  const json = match ? match[1].trim() : input.trim();
  const plan = JSON.parse(json);

  const idMap = {}; // maps plan IDs to Notion page IDs

  /* Pass 1: create all top-level tasks */
  for (const task of plan.tasks) {
    const pageId = await createTask({
      title: task.title,
      description: task.description,
      status: "todo",
      provider: task.provider,
      priority: task.priority,
      project: plan.project,
      tags: task.tags,
    });
    idMap[task.id] = pageId;
    console.log(`Created: ${task.title} (${pageId})`);
  }

  /* Pass 2: create subtasks */
  for (const task of plan.tasks) {
    if (!task.subtasks) continue;
    for (const subtask of task.subtasks) {
      const pageId = await createSubtask(idMap[task.id], {
        title: subtask.title,
        description: subtask.description,
        status: "todo",
        provider: subtask.provider,
        priority: subtask.priority,
        project: plan.project,
      });
      idMap[subtask.id] = pageId;
      console.log(`  Subtask: ${subtask.title} (${pageId})`);
    }
  }

  /* Pass 3: create relations (blocked_by, related) */
  for (const task of plan.tasks) {
    if (task.blocked_by) {
      for (const blockerId of task.blocked_by) {
        if (idMap[blockerId]) {
          await createRelation(idMap[task.id], "Blocked By", idMap[blockerId]);
          console.log(`  ${task.id} blocked by ${blockerId}`);
        }
      }
    }
    if (task.related) {
      for (const relatedId of task.related) {
        if (idMap[relatedId]) {
          await createRelation(idMap[task.id], "Related To", idMap[relatedId]);
        }
      }
    }
  }

  console.log(`\nDone. Created ${Object.keys(idMap).length} tasks.`);
}

main().catch(console.error);

src/providers.js

const { spawn } = require("child_process");
const config = require("../config.json");

function spawnAgent(provider, prompt, cwd) {
  const adapter = config.providers[provider];
  if (!adapter) {
    throw new Error(`Unknown provider: ${provider}`);
  }

  const args = [
    adapter.prompt_flag, prompt,
    adapter.cwd_flag, cwd,
    ...adapter.extra_args,
  ];

  console.log(`[dispatcher] Spawning ${adapter.command} ${args.join(" ")}`);

  const child = spawn(adapter.command, args, {
    cwd,
    stdio: ["pipe", "pipe", "pipe"],
    env: { ...process.env },
  });

  let stdout = "";
  let stderr = "";

  child.stdout.on("data", data => { stdout += data; });
  child.stderr.on("data", data => { stderr += data; });

  return new Promise((resolve) => {
    const timeout = setTimeout(() => {
      child.kill("SIGTERM");
      resolve({ exitCode: -1, stdout, stderr: stderr + "\n[timeout]" });
    }, adapter.timeout_seconds * 1000);

    child.on("close", (code) => {
      clearTimeout(timeout);
      resolve({ exitCode: code, stdout, stderr });
    });
  });
}

module.exports = { spawnAgent };

src/poller.js

const {
  queryInProgressUnassigned,
  getTaskDescription,
  markAssigned,
  markComplete,
  getBlockedByTasks,
} = require("./notion-client");
const { spawnAgent } = require("./providers");
const config = require("../config.json");

const activeJobs = new Set();

async function tick() {
  if (activeJobs.size >= config.poll.max_concurrent) {
    console.log(`[poller] Max concurrency reached (${activeJobs.size}), skipping`);
    return;
  }

  const tasks = await queryInProgressUnassigned();
  console.log(`[poller] Found ${tasks.length} tasks ready`);

  for (const task of tasks) {
    if (activeJobs.size >= config.poll.max_concurrent) break;

    /* Check dependencies */
    const blockers = await getBlockedByTasks(task.id);
    const unresolved = blockers.filter(b => b.status !== "done");
    if (unresolved.length > 0) {
      console.log(`[poller] ${task.id} blocked by ${unresolved.length} unresolved tasks, skipping`);
      continue;
    }

    /* Read task details */
    const title = task.properties.Name.title[0]?.plain_text || "Untitled";
    const provider = task.properties.Provider.select?.name || config.default_provider;
    const description = await getTaskDescription(task.id);
    const prompt = description || title;

    /* Mark as assigned (prevents double-dispatch) */
    await markAssigned(task.id);
    activeJobs.add(task.id);

    console.log(`[poller] Dispatching "${title}" to ${provider}`);

    /* Spawn agent (async, non-blocking) */
    spawnAgent(provider, prompt, config.repo.path)
      .then(({ exitCode, stdout, stderr }) => {
        console.log(`[poller] Task ${task.id} finished with exit code ${exitCode}`);
        return markComplete(task.id, exitCode, stderr);
      })
      .then(() => {
        activeJobs.delete(task.id);
      })
      .catch(err => {
        console.error(`[poller] Error on task ${task.id}:`, err);
        activeJobs.delete(task.id);
      });
  }
}

function start() {
  console.log(`[poller] Starting daemon. Poll interval: ${config.poll.interval_seconds}s, max concurrent: ${config.poll.max_concurrent}`);
  tick();
  setInterval(tick, config.poll.interval_seconds * 1000);
}

start();

11. Limitations & Workarounds

Limitation Impact Workaround
Notion has no webhooks for database changes Can't get push notifications when a task moves to in_progress Poll every 30s (free, ~30s max delay). Acceptable for this workflow. Alternatively, use a Make.com/Zapier bridge that watches Notion and hits a local webhook, but that adds cost and complexity.
Notion API rate limit: ~3 req/s Large task plans (50+ tasks) might hit rate limits during population Add a small delay (350ms) between API calls in populate.js. The populator already handles this implicitly through async/await, but add an explicit sleep for safety.
Each provider CLI has different flags Can't use a single command for all providers The provider adapter map in config.json handles this. Each provider has its own command template. Adding a provider is a config change, not a code change.
Agent output may be very long Notion page body has a 2000-char block limit Truncate output to 2000 chars in markComplete() (already done in the code). For full logs, write to a local file and add a "see logs/task-.log" note.
No guarantee the agent succeeds Agent might produce broken code or fail to complete the task On failure (exit code ≠ 0), the ticket goes back to todo with the error log attached. You can review, adjust the description, and move it back to in_progress to re-dispatch.
Concurrent agents might conflict Two agents working on the same repo simultaneously might step on each other Set max_concurrent: 1 in config if this is a concern. Or use git branches per task (the dispatcher can create a branch before spawning the agent). This is a future enhancement.
Breakdown JSON must be well-formed If the LLM produces malformed JSON, the populator fails Wrap the JSON in <task-plan> tags for reliable extraction. Add a JSON validation step in populate.js with a helpful error message. Optionally, use a cheap LLM (Gemini Flash) as a fallback parser if direct JSON parsing fails.

Summary

This system turns your CLI prompt into a full project management workflow in Notion, with automatic multi-provider dispatch, for $0 in token overhead. The only tokens you spend are on the initial breakdown (which you're already doing) and the actual task execution (which is the real work). All project management — ticket creation, status tracking, dependency management, state updates — runs on free Notion API calls from deterministic scripts.
